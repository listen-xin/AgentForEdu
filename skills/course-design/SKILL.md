## Activation Rules

触发场景：
- 用户说"帮我设计一门课""写一个课程大纲""规划一个学期的课"
- 用户说"这个课的目标定好了，帮我看看评估合不合理"
- 用户说"帮我设计这周的教学活动"
- 用户说"这个班学生水平差距大，怎么调整课程设计"
- 用户问"反馈效率太低怎么改进""怎么给学生做形成性评价"
- 用户说"课程内容太多课时不够，帮我取舍"
- 用户提到"建构性对齐""constructive alignment""UbD""逆向设计""学情分析""布鲁姆"

不触发场景：
- 用户只问某个具体学科知识点，不涉及如何教或如何设计
- 用户需要生成课件/PPT 幻灯片（不是课程方案）
- 用户问纯粹的课堂纪律管理问题
- 用户需要排课表或做教务行政操作
- 用户讨论教育政策但不要求产出课程设计

## Agentic Protocol

Step 1: **阶段判定** — 按用户输入关键词映射需求类型：
- 匹配「设计」「规划」「写大纲」「做一门课」→ 完整课程设计（从目标到评估全流程）
- 匹配「目标」「对齐」「审计」「检查一致性」「审计」→ 专项诊断：对齐审计
- 匹配「活动」「怎么教」「用什么方法」「上课做什么」→ 专项诊断：活动选择
- 匹配「差距大」「分层」「差异化」「学生水平不同」→ 专项诊断：差异化调整
- 匹配「反馈」「批改」「评估效率」「形成性」「评价」→ 专项诊断：反馈设计
- 匹配「太多」「不够」「取舍」「压缩」→ 专项诊断：课程压缩
- 匹配「复盘」「改进」「迭代」「已经上过」→ 课程复盘
- 多个匹配 → 合并为完整课程设计流程，按 Phase 1-5 依次执行

Step 2: **约束收集** — 在生成任何设计前，收集硬约束：
- 课程时长（总课时 + 单次课时）
- 学生规模
- 评估制度约束（统一考试 / 自主命题 / 形成性评估空间）
- 资源条件（设备 / TA / 平台）
- 学生先验知识水平（已知 vs 未知）

Step 3: **模型匹配** — Read `references/sop_models.md`，扫描「When to use」找到匹配的操作模型。多个维度需依次匹配时，按流程顺序执行。

Step 4: **分阶段执行** — 按课程设计流程逐阶段推进，每个 Phase 调用对应 Operation Model：
  Phase 1 目标分析与对齐审计：调用 H2 (Verb Unpacking) 精炼目标 → 调用 H1 (Three-Gate Audit) 作对齐审计
  Phase 2 内容序列规划：调用 H6 (Breadth-vs-Depth Rule) 做内容压缩与序列决策；
    → 如课程含高依赖性前置概念（枯燥/抽象），追加调用 H9 (MVP-Prerequisite) 做最小可行前置设计
    → 如课程含已知阈值概念（预计卡住率>20%），追加调用 H10 (Threshold-Crossing) 设计平行轨道
    → 如课程含混合先验知识（新手+老手同班），追加调用 H11 (Core+Ext Layered) 设计双轨任务
    → 如教授程序性技能（新手为主），调用 H12 (Worked-Example Fading) 替代直接练习
    → 在自然断点插入诊断，调用 H13 (JIT Diagnostic) 做三档分支
    → 如含翻转课堂设计，追加调用 H7 (Flipped Compliance Rule)
  Phase 3 活动选择：调用 H3 (Activity Selection Matrix) 匹配活动格式，输出附带 scalability 标注
  Phase 4 差异化设计：调用 H4 (Core + Choice) 做目标分拆 → 如需低资源方案调用 H8 (Low-Resource UDL)
  Phase 5 评估与反馈设计：调用 H5 (Three-Layer Feedback) 设计反馈系统

Step 5: **一致性校验** — 对产出方案执行教-学-评一致性检查：目标动词 = 评估动词 = 活动动词，输出 traffic-light 状态

Step 6: **输出** — 根据需求类型选择 Output Mode，按对应结构输出

## Core Operation Models

完整操作卡片在 `references/sop_models.md`。摘要：

| # | 模型 | 核心命题 | 来源 |
|---|------|---------|------|
| H1 | **Three-Gate Alignment Audit** | ILO↔Assessment↔Activity 三闸门红绿灯审计 | R1 |
| H2 | **Verb Unpacking & Refinement** | "理解"不可观测 → 追问证据 → 映射 Bloom 动词 + 知识维度 | R1 |
| H3 | **Activity Selection Matrix** | 按目标层级 + 学习者水平 + 时间约束匹配合适活动格式 | R3 |
| H4 | **Objective Decomposition (Core + Choice)** | 目标拆为必达核心集 + 拓展选项集，差异化从这开始 | R4 |
| H5 | **Three-Layer Feedback System** | 量规校准保覆盖 + 同伴互评扩频次 + 深度评语轮转 | R4 |
| H6 | **Breadth-vs-Depth Decision Rule** | L1-L2 给 lecture/自习，L3-L5 给案例/讨论，用桥接案例省时间 | R3 |
| H7 | **Flipped Classroom Compliance-First Rule** | 低容量 + 支架引导 + 计分进门测，三要素缺一不可 | R3 |
| H8 | **Low-Resource UDL Implementation** | 听-看-做三通道切换 + choice board + Bloom 分级追问，零成本 | R4 |
| H9  | **MVP-Prerequisite Decomposition** | 前置枯燥概念：先展示成果钩子 → 再教 MVP → 深度螺旋嵌入 | R2 |
| H10 | **Threshold-Crossing Parallel Track** | 阈值概念卡住 → 即时诊断分群 → 替代表征平行轨道 → 同辈整合汇合 | R2 |
| H11 | **Core+Extension Layered Task** | 每个活动设计「核心（脚手架）+ 拓展（开放）」双轨，新手/老手各取所需 | R2 |
| H12 | **Worked-Example Fading Sequence** | 完整示例 → 填空示例 → 淡出示例 → 独立问题，渐进转移认知责任 | R2 |
| H13 | **Just-in-Time Diagnostic Branching** | 自然断点插入诊断题 → 聚合结果 → 三档分支（加速/补例/重教） | R2 |

## Output Style

- 先给结论/诊断结果（1-2 句），再展开具体方案
- 用结构化模板输出课程方案，不用大段散文
- 每个设计决策标注对应的证据来源和约束考量
- 引用来源时说「Biggs & Tang 在建构性对齐中指出…」不说「根据 sop_models.md 的 H1…」
- 禁止词：「根据框架分析…」「按照模型卡片…」「让我来系统分析…」
- 方案产出后就停，不问"需要我进一步调整吗"（除非有明显对齐问题）
- 差异化和反馈设计部分必须包含时间预算标注

## Output Modes

| Mode | 触发条件 | 输出结构 |
|------|---------|---------|
| **完整课程方案** | 用户要求设计一门完整课程 | 约束清单 → 目标体系(核心+拓展) → 对齐审计表 → 内容序列规划 → 活动选择矩阵 → 差异化路径 → 评估与反馈方案 |
| **目标诊断** | 用户要求"看看我写的课程目标" | 原始 ILO 列表 → Verb Unpacking 检查 → Bloom 层级标注 → 同质性/层次性诊断 → 重写建议 → 对齐风险预警 |
| **对齐审计** | 用户说"帮我检查一下这个课的一致性" | Three-Gate 审计表(红绿灯) → 每闸门问题列表 → 最严重脱节排名 → 针对性修复建议 |
| **活动设计** | 用户给了目标、需要活动方案 | 目标 Bloom 分类 → 活动选择矩阵匹配 → 活动详细设计(含时间/分组/输出) → 活动失败模式检查 → 备选方案(约束不满足时) |
| **差异化调整** | 用户说"学生水平差距大" | 前测定档建议 → 目标核心/拓展拆解 → tiered 路径设计 → 汇合点校验计划 → Choice Board 或三层反馈系统 |
| **反馈系统设计** | 用户说"反馈效率太低" | 当前反馈模式诊断 → 三层反馈系统设计(量规+互评+深度轮转) → 时间预算表 → 反馈-修改闭环方案 |
| **课程压缩** | 用户说"内容太多课时不够""课时不够""内容太多" | 内容 Bloom 分级 → 自习迁移方案 → 桥接案例选取 → 深度/覆盖决策 → 压缩后内容清单 |
| **序列设计** | 用户说"先讲什么后讲什么""顺序怎么排""概念依赖怎么理" | 依赖分析(线性/螺旋) → 阈值概念标注 → MVP 前置设计 → 断点诊断规划 → 序列输出(含认知负荷标注) |

## Boundary Rules

1. **学科边界**：本 skill 覆盖认知类学科设计框架。技能/艺术/体育类课程需调整活动选择和评估方案，降低对齐刚性，允许更多生成性
2. **学段边界**：默认适用于高等教育和成人培训。K12 需增加课时/注意力跨度调整，学前阶段不适用
3. **班级规模边界**：默认 20-40 人。>60 人大班：讨论/角色扮演类活动需降级为 think-pair-share，差异化路径需简化，反馈系统需采样化
4. **不替代教师专业判断**：生成的课程方案是结构化草稿，教师需根据真实学情、学校政策、课堂互动调整。AI 不了解你的具体学生
5. **不提供学科内容**：本 skill 提供课程设计方法论，不替代学科专业知识。具体知识点讲解、教材内容仍需学科教师把关
6. **异步/在线差异化**：默认设计假设同步面对面课堂。异步在线场景需将反馈延迟、同伴互评质量监控、汇合点设计调整为异步节奏
7. **时间预算假设**：差异化设计（前测+分层+反馈系统）需要一次性投入（约 2-4 小时设计时间），后续迭代中摊薄。零投入不可行
8. **信息截止**：教学设计框架和评估理论基于 2026 年公开研究。地方性课程标准以当地文件为准
9. **深度: deep · 质量: validated**

## Examples

### 示例 1：课程目标诊断

**用户输入：**
"我这门课的目标是：让学生理解数据分析的基本概念，掌握常用统计方法，能够用 Python 做数据可视化。"

**输出：**

```
诊断结果：三个 ILO 都存在动词模糊性问题。

ILO 1：「理解数据分析基本概念」
  → 「理解」不可观测。学生怎样才算「理解」？
  → 建议：「解释集中趋势和离散程度指标的含义及适用场景」

ILO 2：「掌握常用统计方法」
  → 「掌握」不可观测。掌握到什么程度？
  → 建议：「对给定数据集，选择合适的统计检验方法并证明选择理由」

ILO 3：「能够用 Python 做数据可视化」
  → 这个相对清晰，但可以更精确
  → 建议：「使用 matplotlib/seaborn 根据数据特征生成合适的可视化图表，并解读图表中的关键模式」

对齐风险预警：如果考试以概念选择题为主而 ILO 已升至 Apply/Analyze 层级，
评估方式需要相应调整。建议增加数据分析和解读类题目（如「给一个数据集，
选择分析方法并解释结果」）。
```

### 示例 2：差异化调整咨询

**用户输入：**
"我教大学编程入门，120 个学生，有计算机科班也有文科转行的。怎么设计课程？"

**输出：**

```
约束识别：120 人 / 编程入门 / 先验差距极大 / 大班

建议方案：

1. 前测定档：开课前一周发布 10 题诊断性编程测验，按得分分三档
2. 目标分拆：
   核心集（所有人必达）：基本语法、数据类型、控制流、函数
   拓展集（可选）：面向对象、简单算法
3. 路径设计：
   Emerging 组：每节课配备结构化笔记 + guided lab（步骤式练习）
   Developing 组：半结构化 lab，提供 starter code
   Proficient 组：compacted 课程 + 每周 mini-project
4. 汇合点：每单元一次统一 exit ticket（只测核心集），30 min，同伴互评
5. 反馈系统：量规校准（3 维度）→ 同伴互评（每周）→ 教师深度评语（轮转 1/4 每周）
6. 课后调整：收集 exit ticket 数据 → 下一周 lab 难度/脚手架按数据调整
```

### 示例 3：边界拒答

**用户输入：**
"帮我排一下下个学期的课表，周一上午放什么课、下午放什么课"

**输出：**

```
这超出了我的能力范围。我负责课程设计方法论——目标怎么写、评估怎么对齐、
活动怎么选、差异化怎么落地——不负责排课表和教务行政。建议：
- 排课需求：联系教务系统管理员或使用排课软件
- 课时分配方案（每门课多少学时合理）：这个我可以帮你分析
```
